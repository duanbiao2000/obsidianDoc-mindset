# 工程决策的通用原则

> **核心思想**：停止收藏“定律”，开始将它们翻译成可执行的、用于日常决策的工程启发式算法。

### 行动纲领：将定律转化为决策算法

-   **林迪效应 (Lindy Effect)**
    -   **启发式**：“核心业务用‘老古董’，边缘创新用‘新玩具’。”
    -   **应用**：技术选型时，为数据库、消息队列等关键组件选择经受过长期考验的技术（如 PostgreSQL），以此规避新技术的隐性风险。

-   **奥卡姆剃刀 (Occam's Razor)**
    -   **启发式**：“先用 `if`，再想多态。”
    -   **应用**：开发新功能时，用最直接的方式实现。拒绝任何在需求明确前就进行的过早抽象，只在模式重复出现时才进行重构。

-   **反脆弱性 (Antifragility)**
    -   **启发式**：“主动注入故障，强制系统进化。”
    -   **应用**：在系统设计评审中，必须回答：“如果这个依赖挂了，系统会优雅降级还是直接崩溃？” 定期通过混沌工程来验证和强化系统的韧性。

-   **复杂性守恒 (Complexity Conservation)**
    -   **启发式**：“简化是有代价的，代价由谁承担？”
    -   **应用**：设计一个极简的 API 意味着复杂性被转移到了后端。必须评估这种转移是否增加了维护成本和不可控的风险。

-   **第一性原理 (First Principles)**
    -   **启发式**：“停止说‘最佳实践’，开始问‘为什么’。”
    -   **应用**：在重构或架构决策时，回归问题的本质。例如，不要问“我们该用什么微服务框架？”，而是问“我们要解决的根本问题是组织扩张快，还是业务逻辑解耦？”

-   **汉隆的剃刀 (Hanlon's Razor)**
    -   **启发式**：“优先假设是愚蠢，而不是恶意。”
    -   **应用**：排查跨团队 Bug 时，首先反思“我们的 API 文档是否有歧义？”，而不是指责“他们又写了烂代码！”。这能将冲突转为协作。

---

### 核心行动：从收藏家到架构师

不要再仅仅是记忆这些原则。在下一次技术评审或决策时，**强制自己用其中一条作为论据**。

> 例如：“根据**林迪效应**，这个新技术对核心业务风险太高。我建议……”

将这些原则编译成你大脑中可执行的决策子程序，是从博学的工程师到顶尖架构师的关键跃迁。
